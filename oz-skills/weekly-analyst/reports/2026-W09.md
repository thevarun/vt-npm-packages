## Improvement Suggestions

### Detailed Suggestions

#### 1. [HIGH] Reduce orchestrator context overhead for fix-audit-findings

**Root Cause:** BMAD workflow architecture requires full step-file reading as a core principle, but the fix-audit-findings orchestrator needs much more context than typical BMAD workflows because it also loads audit artifacts.

## Research Insights

### Context Management for Heavy Subagent Workflows

**Triggered by**: Frequent `/clear` usage and multi-session workflow continuity challenges
**Source**: [MCPcat Context Management Guide](https://mcpcat.io/guides/managing-claude-code-context/), [Claude Platform Context Editing](https://platform.claude.com/docs/en/build-with-claude/context-editing)
**Key takeaway**: Claude's API now supports automatic `clear_tool_uses` context editing that clears old tool results when context grows, which reduced token consumption by 84% in evaluations. Additionally, proactive compaction at 70% capacity (rather than waiting for auto-compact at ~83%) produces higher quality output.
**Relevance to user**: The fix-audit-findings workflow with 12+ subagent calls per session would benefit significantly from automatic tool result clearing. The user's current `/clear` pattern is more aggressive than needed.

### Subagent Orchestration: Task Tool vs Custom Agents

**Triggered by**: Heavy Task tool usage for theme execution in fix-audit-findings
**Source**: [Claude Code Subagents Docs](https://code.claude.com/docs/en/sub-agents), [Shrivu Shankar's Guide](https://blog.sshh.io/p/how-i-use-every-claude-code-feature)
**Key takeaway**: Custom subagents with restricted tool access (e.g., implementation subagents with only Read/Edit/Bash) can prevent accidental state corruption. However, there's a tradeoff: custom subagents gatekeep context from the main agent. The user's approach of using generic Task calls with rich prompts is a good balance.
**Relevance to user**: The BMAD fix-audit-findings workflow already uses Task well. Consider defining a custom subagent for "enrichment" and "implementation" with restricted tools to prevent accidental sidecar edits from implementer subagents.

### Context Optimization via Lazy Loading

**Triggered by**: Large step-file consumption in orchestrator context
**Source**: [Context Optimization Gist](https://gist.github.com/johnlindquist/849b813e76039a908d962b2f0923dc9a)
**Key takeaway**: A developer achieved 54% reduction in initial context by switching from verbose documentation to trigger-based routing — loading detailed protocols on-demand rather than upfront.
**Relevance to user**: The BMAD step-file architecture loads full step files upfront per its "READ COMPLETELY" principle. For the orchestrator role specifically, a condensed version that skips documentation sections could significantly reduce overhead.

## Innovations

### 1. Session State Handoff File for Multi-Session Workflows

**Pattern observed**: The fix-audit-findings workflow spans 4+ sessions. Each new session re-reads multiple files to reconstruct state, consuming ~30-40% of initial context on boilerplate.
**Opportunity**: The sidecar tracks execution state but doesn't include a human/agent-readable "handoff summary."
**Specific proposal**: At the end of each session (or on `/compact`), automatically generate a `handoff.md` file containing: (1) what was completed this session, (2) what's next, (3) key decisions made, (4) any blockers. On resume, the orchestrator reads only `handoff.md` + `sidecar.yaml` instead of re-reading full workflow/step files.
**Expected impact**: 50%+ reduction in context overhead on resume. Faster time-to-first-action in new sessions.

### 2. Enrichment Cache for Repeated Audit Themes

**Pattern observed**: The fix-audit-findings workflow enriches each theme via a subagent before implementation. If a session is interrupted and resumed, enrichment may need to re-run.
**Opportunity**: Enrichment output is already written to `{enrichment_output_folder}/{theme_id}-enrichment.md`, but the workflow doesn't check for existing enrichment before spawning the subagent.
**Specific proposal**: In step-02-execute.md Phase B, add a check: if `{theme_id}-enrichment.md` already exists and is non-empty, skip enrichment and proceed directly to implementation. This is an idempotency optimization.
**Expected impact**: Saves 1-2 minutes per theme on resume, reduces subagent context costs.

### 3. Cross-Workflow Learning: Apply Session Sync Patterns to Audit State

**Pattern observed**: In vt-spells, the user planned a git-based sync mechanism for session data. In vt-saas-template, the audit-fix state is tracked in a YAML sidecar.
**Opportunity**: The session sync approach (automated cron + git) could be adapted for audit-fix state, enabling cross-device or cross-session visibility.
**Specific proposal**: Auto-commit the sidecar state file after each theme completion (the workflow already commits code changes per theme). This creates a git history of execution progress that can be inspected from any session or device.
**Expected impact**: Better observability of workflow progress without needing an active Claude session.
